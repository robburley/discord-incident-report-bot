# Repository Guidelines

## Project Structure & Module Organization

This is a TypeScript Cloudflare Workers Discord bot. The Worker entrypoint is `src/platform/cloudflare.ts`. Keep platform-neutral business logic in `src/core`, Discord command and interaction code in `src/discord`, database schemas and repositories in `src/db`, and Cloudflare-specific bindings or adapters in `src/platform`. Command registration utilities live in `scripts`. D1 schema migrations are stored in `migrations` and generated by Drizzle. Tests mirror the source layout under `tests`, such as `tests/core` and `tests/discord`.

## Build, Test, and Development Commands

- `npm install`: install project dependencies.
- `npm run dev`: start the Worker locally with Wrangler.
- `npm run typecheck`: run `tsc --noEmit` with strict TypeScript checks.
- `npm test`: run the Vitest suite once.
- `npm run test:watch`: run Vitest in watch mode while developing.
- `npm run db:generate`: generate Drizzle migration files.
- `npm run db:migrate`: apply D1 migrations to the local Wrangler database.
- `npm run register:commands:guild`: register slash commands to the configured test guild.
- `npm run deploy`: deploy with `wrangler.prod.toml`.

## Coding Style & Naming Conventions

Use ES modules and TypeScript throughout. The project is configured with strict compiler options, including `noUncheckedIndexedAccess` and `exactOptionalPropertyTypes`; prefer explicit validation and narrow types over casts. Use two-space indentation, double quotes, and semicolons only where already needed by surrounding code. Name source files in lowercase kebab case, for example `interaction-handler.ts`, and keep test files named `*.test.ts`.

## Testing Guidelines

Vitest is the test framework, running in a Node environment with globals enabled. Place tests under `tests/**/*.test.ts`, grouped by feature or layer. Add or update tests for command behavior, interaction handling, database repository changes, and security-sensitive request validation. Run both `npm test` and `npm run typecheck` before opening a pull request.

## Commit & Pull Request Guidelines

Recent commits use short imperative summaries, such as `Add rate limiting` and `Remove test logging`. Keep commit subjects concise and action-oriented. Pull requests should describe the behavior change, list the verification commands run, and note any migration, secret, Wrangler, or Discord command-registration impact. Include screenshots only when Discord-facing output or user-visible formatting changes.

## Security & Configuration Tips

Do not commit `.dev.vars` or real Discord tokens. Use `.dev.vars.example` as the local template. Production secrets should be set with the provided `secrets:set:*` npm scripts. Keep the D1 binding name aligned with `INCIDENT_DB` in Wrangler configuration.
